# Plum OPD Claim Adjudicator

AI-powered tool that automates the **adjudication (approve / reject / partial / manual-review)** of
Outpatient Department (OPD) insurance claims — built for the Plum AI Automation Engineer intern assignment.

It ingests medical documents (or structured claim data), extracts the key fields, validates them
against the policy terms, applies the adjudication rules, and returns a decision with **approved amount,
reasons, confidence score, next steps, and a full audit trail**.

> **Engine accuracy: 100%** on all 10 provided test cases (decision + amount + reason). Run `npm run eval` to verify.

---

## ✨ Highlights

| Area | What it does |
|------|--------------|
| **Hybrid decision engine** | A deterministic TypeScript rule engine owns the decision (fast, reproducible, testable); the LLM adds reasoning + a medical-necessity guardrail but **never silently changes outcomes**. |
| **AI document extraction** | Upload a prescription/bill image or PDF → **Gemini vision** extracts structured fields → review → adjudicate. |
| **RAG grounding** | Each decision retrieves the most relevant policy/rule clauses (shown as "policy basis") and feeds them to the LLM for grounded explanations. |
| **Confidence + guardrails** | Every decision carries a confidence score; low-confidence and fraud-flagged claims auto-route to **manual review**. |
| **Eval harness** | `npm run eval` (CLI) and an in-app **Evals dashboard** score the engine against expected outputs — decision / amount / reason accuracy + confidence delta. |
| **Graceful degradation** | Runs **fully without an API key** (deterministic engine only), so it's always demoable. The LLM layer turns on automatically when `GEMINI_API_KEY` is set. |
| **Admin policy view** | The whole policy is rendered from `policy_terms.json`; editing that file reconfigures the engine. |

---

## 🏗️ Tech stack

- **Next.js 14 (App Router) + TypeScript** — React front-end **and** serverless API routes in one Vercel-deployable app.
- **Google Gemini API** (`@google/genai`) — multimodal vision extraction + reasoning.
- **Tailwind CSS** — UI.
- **RAG-lite** — in-process token-overlap retrieval over the rules/policy corpus (no vector DB needed; clean upgrade path documented).
- **JSON document store** — zero-dependency structured persistence (swappable for Postgres/Supabase).

## 🚀 Quick start

```bash
cd plum-opd-adjudicator
npm install

# (optional) enable the AI layer
cp .env.example .env        # then add your GEMINI_API_KEY

npm run dev                 # http://localhost:3000
```

Without a key the app still runs end-to-end on the deterministic engine. With a key, document upload and
AI reasoning light up automatically (the header shows **AI layer: ON/OFF**).

### Useful commands

```bash
npm run eval        # run all test cases through the engine, print a scoreboard (exits non-zero on failure → CI gate)
npm run gen:docs    # (re)generate the sample medical-document PDFs in ./sample-documents
npm run typecheck   # tsc --noEmit
npm run build       # production build
```

## 🖥️ Using the app

1. **Adjudicate** (home) — three input modes:
   - **Sample cases** — one-click load any of the 10 provided test cases.
   - **Claim data** — editable JSON (the `input_data` shape from `test_cases.json`).
   - **Upload docs** — images/PDFs → Gemini extracts → review → adjudicate *(needs API key)*.
2. **Claims** — history of every adjudicated claim with its decision and drill-down.
3. **Evals** — run the suite and see decision/amount/reason accuracy live.
4. **Policy** — the active policy configuration the engine adjudicates against.

## 🔌 API

| Method | Route | Purpose |
|--------|-------|---------|
| `POST` | `/api/adjudicate` | Adjudicate a claim (`input_data` shape). Returns decision + result. |
| `POST` | `/api/extract` | Multipart upload of document images/PDFs → structured claim (needs API key). |
| `GET`  | `/api/claims` | List stored claims. |
| `GET`  | `/api/evals` | Run the test suite, return accuracy metrics. |

Example:

```bash
curl -s http://localhost:3000/api/adjudicate \
  -H "Content-Type: application/json" \
  -d '{"member_name":"Rajesh Kumar","treatment_date":"2024-11-01","claim_amount":1500,
       "documents":{"prescription":{"doctor_reg":"KA/45678/2015","diagnosis":"Viral fever"},
       "bill":{"consultation_fee":1000,"diagnostic_tests":500}}}'
```

## 🧠 How a decision is made (in one paragraph)

`normalize` raw input into a canonical claim → `retrieve` relevant policy clauses (RAG) →
the **rule engine** evaluates rules in priority order (fraud → documents → eligibility → process →
coverage → limits → settlement) and computes the approved amount (co-pay vs network discount) →
the **LLM** (if enabled) writes a grounded explanation, calibrates confidence, and can escalate to
manual review on a medical-necessity concern → **guardrail** routes any sub-0.70-confidence decision to
manual review → result is stored and returned. See [`docs/DECISION_LOGIC.md`](docs/DECISION_LOGIC.md).

## ☁️ Deployment

Deploy to **Vercel** (zero config): import the repo, add `GEMINI_API_KEY` as an env var, deploy.
Note: the JSON claim store resets on serverless cold starts — wire up Postgres/Supabase for durable
production storage (the `src/lib/db/store.ts` interface is drop-in swappable). See `docs/ASSUMPTIONS.md`.
